# Difficulty Levels

## Purpose
Difficulty in this repo is defined by reasoning load, pattern obviousness, implementation burden, and edge-case handling, not by brand-name problem reputation alone.

## Easy
### Reasoning Load
Low. The path from problem statement to approach is short.

### Pattern Obviousness
Usually obvious after a careful read. The problem often points clearly toward one standard pattern.

### Implementation Burden
Light. The code is short and the state is easy to track.

### Edge-Case Handling
Limited. There are still edge cases, but they are usually straightforward.

### Typical Signal
The main challenge is careful execution, not discovering a hidden idea.

## Medium
### Reasoning Load
Moderate. The user usually needs to compare at least two approaches and justify the better one.

### Pattern Obviousness
Visible, but not immediate. The pattern often appears only after analyzing constraints or walking a small example.

### Implementation Burden
Moderate. The code may involve multiple moving parts, tracked state, or a more delicate invariant.

### Edge-Case Handling
Meaningful. Missing one boundary condition can break an otherwise correct approach.

### Typical Signal
The user can solve it with solid pattern recognition and disciplined implementation, but not by reflex alone.

## Hard
### Reasoning Load
High. The user must define state carefully, choose among multiple plausible strategies, or derive a non-obvious reduction.

### Pattern Obviousness
Low. The pattern is often hidden, mixed with another pattern, or easy to misidentify at first.

### Implementation Burden
High. The solution may require careful recursion, multi-stage reasoning, dynamic programming state, or subtle invariant management.

### Edge-Case Handling
Heavy. Multiple failure modes exist, and correctness often depends on precise handling of special conditions.

### Typical Signal
The problem tests whether the user can stay organized when the reasoning path is not obvious and the implementation has little slack for mistakes.
